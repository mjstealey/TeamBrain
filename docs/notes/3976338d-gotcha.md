## Content

**Gotcha — seeding `public.app_config` for the membership-sync cron: a wrong value silently overrides the working GUC; seed FROM the GUC, not by re-pasting from .env.**

Migration 0013 moved the sync cron's URL + service-role bearer from per-database GUCs into `public.app_config`, read as `coalesce((select value from public.app_config where key=…), current_setting('app.teambrain_*', true))`. Two operational traps learned at deploy (2026-06-01):

1. **coalesce prefers app_config.** Once an app_config row exists, it WINS over the GUC. So a wrong/truncated app_config value overrides the working GUC and breaks the next */15 fire — the gateway's VERIFY_JWT rejects a non-JWT bearer and NO `sync_runs` row is written (silent — exactly the failure the new healthcheck catches). During this deploy a 34-char mis-paste did precisely that: the 14:15–14:45 fires were rejected, the */30 healthcheck logged a `health_events` stale row, and it recovered at 15:00 after the fix.

2. **Seed the secret FROM the GUC, not by re-pasting .env.** The service_role key is a long (~180-char) HS256 JWT; re-pasting from .env risks truncation / wrong line (the 34-char value above). The GUC already holds the known-good value (the pre-0013 cron used it successfully), so copy it across in SQL:
```
insert into public.app_config (key, value)
values ('teambrain_service_role_key', current_setting('app.teambrain_service_role_key', true))
on conflict (key) do update set value = excluded.value, updated_at = now();
```
Then verify `length(value)` ≈ 180 (not 34). Do NOT grab `SERVICE_ROLE_KEY_ASYMMETRIC` from .env — it's the empty ES256 var; the stack is legacy HS256 (see the separate context thought).

**After 0013, always:** `NOTIFY pgrst, 'reload schema';` (so PostgREST exposes app_config + membership_sync_health to the function's service client), then confirm `GET /functions/v1/teambrain-membership-sync/health` returns 200 `ok` and `config.service_role_key_set: true`.
